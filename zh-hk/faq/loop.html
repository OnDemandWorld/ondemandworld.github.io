01 · 概覽

基本概覽 — Loop 是什麼？

9 個問題

Loop 是 ODW.ai 套件的編排層。它將 ODW 的各個代理（Vault、Desk、Recap 等）串連成自動化的多步驟工作流，全部在你的基建上運行。

運作方式如下：

  1. 視覺化建構器：拖放節點建立工作流（無需編碼）
  2. 代理元件：使用預設節點處理 Vault（知識庫）、Desk（任務）、Recap（會議情報）
  3. 觸發方式：工作流可按排程、事件或手動觸發
  4. 執行：Loop 負責編排工作流，在代理之間傳遞數據

工作流範例：「會議結束後 → 提取行動項目（Recap）→ 建立任務（Desk）→ 更新知識庫（Vault）」

與其他工作流工具的最大分別：代理原生、主權設計、模型無關。Loop 將 ODW 的代理元件視為一等物件，所有數據留在你的基建上。

企業在 AI 代理上面臨多重挑戰：

  • 代理各自為政：各個代理（知識庫、會議情報、任務管理）獨立運作，缺乏協調
  • 雲端依賴：大部分工作流工具（Zapier、n8n）需要將數據傳送至其雲端，構成合規風險
  • 供應商綁定：工作流工具綁定特定 AI 模型或雲端供應商
  • 複雜性：建立多代理工作流需要深厚技術專長

Loop 以代理原生編排配合主權設計解決這些問題。你可以在無需雲端依賴或供應商綁定的情況下，將 ODW 代理協調成生產工作流。

Loop 專為受監管或對數據駐留敏感的中小企而設：

  • 法律：律師樓、法律科技——律師與客戶保密特權
  • 醫療：醫院、診所、醫療科技——類 HIPAA 工作流
  • 金融服務：銀行、金融科技、保險——嚴格的數據主權
  • 政府承辦商：國防、聯邦——嚴格的數據主權
  • 注重私隱的企業：歐盟、亞太、拉美地區——GDPR 合規
  • 任何企業：希望在無雲端依賴下自動化 AI 代理工作流

如果你需要協調 AI 代理並同時確保所有數據留在你的基建上，Loop 正是為你而設。

Zapier 和 n8n 是通用工作流工具。Loop 是代理原生編排器。

如果你需要在完全掌控數據的前提下編排 AI 代理，Loop 是更佳選擇。

「代理原生」代表Loop 將 ODW 的代理元件視為一等工作流物件：

  • Vault 文件：Loop 可以讀寫文件、搜尋、執行 RAG 查詢、管理標籤
  • Desk 任務：Loop 可以建立／更新任務、管理項目、進行日曆操作
  • Recap 逐字稿：Loop 可以匯入逐字稿、提取行動項目、摘要、分類

不同於通用工作流工具（將代理視為不透明的 webhook），Loop 理解代理輸出的語意類型。來自 Vault 的「文件」是類型化物件，而非僅是 JSON 資料塊。

為何重要代理原生編排實現類型安全工作流、更佳錯誤處理，以及更直觀的工作流編寫。

Loop 兩者皆是：

  • 獨立使用：你可以單獨使用 Loop 進行工作流自動化
  • 套件整合：Loop 是 ODW.ai 套件的編排層，並與其他模組整合：
    • Vault：知識庫（文件、RAG、標籤）
    • Desk：工作區（任務、項目、日曆）
    • Recap：會議情報（逐字稿、行動項目、摘要）
    • Shield：管治（審計日誌、合規報告）

套件整合正是 Loop 強大的原因——它可以憑完整語意理解將 ODW 代理協調成生產工作流。

「主權設計」代表所有工作流數據留在你的基建上：

  • 零數據外洩：沒有數據離開你的基建（除非你明確設定）
  • 本機執行：所有工作流在你的硬件上執行
  • 無遙測：不會將使用數據傳送至 ODW.ai（除非你選擇加入）
  • 氣隙隔離：Loop 可在氣隙環境中運行（無需互聯網存取）

這對受監管行業至關重要。如果你的工作流工具正將數據傳送至雲端供應商，你就無法聲稱擁有數據主權。

部署方面：需要。Loop 為自託管，因此你需要有人能管理 Docker/Kubernetes、設定網絡及監察基建。

工作流編寫方面：不需要。Loop 提供視覺化拖放建構器——非工程人員可在 30 分鐘內組合工作流。

進階邏輯方面：需要。Loop 提供代碼節點（Python/TypeScript）處理自訂邏輯——工程師可在不離開平台的情況下編寫代碼。

視情況而定。

  • 如需 AI 代理編排：可以。Loop 專為協調 AI 代理（Vault、Desk、Recap 等）而設
  • 如需通用自動化：或許。Loop 可透過 webhook 整合外部服務，但在非 AI 用途上不及 Zapier/n8n 廣泛

Loop 無意取代 Zapier/n8n 的通用自動化角色。它是ODW 主權代理套件的編排器。

如果你已在使用 ODW 代理並需要協調它們，Loop 是正確選擇。如果你需要自動化非 AI 工作流（例如「提交表單後發送電郵」），Zapier/n8n 可能更合適。

02 · 主權

工作流編寫

8 個問題

Loop 提供視覺化拖放建構器：

  1. 節點庫：瀏覽預設節點（Vault、Desk、Recap、邏輯、代碼）
  2. 拖放：將節點拖到畫布上
  3. 連接節點：在節點之間繪製連線以定義數據流向
  4. 設定節點：設定參數、條件及錯誤處理
  5. 測試：在測試模式運行工作流
  6. 部署：啟動工作流投入生產

目標：非工程人員可在 30 分鐘內建立生產工作流。

Loop 支援多種節點類型：

  • 代理節點：Vault（讀寫文件、搜尋、RAG）、Desk（任務、項目、日曆）、Recap（逐字稿、行動項目、摘要）
  • 邏輯節點：條件分支、迴圈、並行執行、錯誤處理
  • 代碼節點：Python 及 TypeScript 自訂邏輯
  • 觸發節點：排程（cron）、事件驅動（webhook、檔案監視）、手動
  • 整合節點：HTTP 請求、數據庫查詢、檔案操作

節點庫可擴展——你可以為特定用途建立自訂節點。

可以。Loop 提供條件分支：

  • If/else：根據條件分支（例如「如情緒為負面，升級至經理」）
  • Switch：根據值進行多向分支
  • 篩選：根據條件篩選數據
  • 迴圈：迭代清單（例如「為每個行動項目建立任務」）

條件邏輯可實現具動態行為的複雜工作流。

可以。Loop 提供代碼節點：

  • 語言：Python 及 TypeScript
  • 用途：自訂數據轉換、複雜邏輯、API 整合
  • 沙盒：代碼在安全沙盒中運行
  • 存取工作流上下文：代碼可以讀寫工作流變量

代碼節點在視覺節點不足時非常有用——工程師可在不離開平台的情況下編寫代碼。

可以。Loop 提供基於 git 的版本控制：

  • 自動提交：每次工作流變更都會提交至 git
  • 分支：為實驗性工作流建立分支
  • 回滾：還原至先前版本
  • 協作：多名用戶可共同處理工作流（附衝突解決）

版本控制確保你能追蹤變更、安全協作，並在出現問題時回滾。

有。Loop 提供預設工作流範本：

  • 會議 → 行動項目 → 任務：從會議逐字稿提取行動項目，在 Desk 建立任務
  • 文件 → 知識庫：匯入文件、提取關鍵資料、更新 Vault
  • 客戶查詢 → 回覆：將客戶查詢路由至合適代理，產生回覆

範本只是起點——你可按特定用途自訂。

可以。Loop 提供測試模式：

  • 模擬運行：以範例數據執行工作流
  • 逐步執行：逐節點執行工作流
  • 除錯：檢視每個節點的中間輸出
  • 日誌：檢視執行日誌以排查問題

測試模式確保你可以在部署至生產前驗證工作流。

可以。Loop 提供子工作流：

  • 模組化設計：建立可重用的工作流元件
  • 參數：將輸入傳遞至子工作流

  • 庫：建立可重用元件庫

子工作流實現模組化、可維護的工作流設計——建立一次，到處重用。

04 · 安全

執行與觸發

8 個問題

Loop 支援多種觸發類型：

  • 排程：Cron 風格排程（例如「每日上午 9 時」、「每小時」）
  • 事件驅動：Webhook、檔案監視器、ODW 代理事件
  • 手動：介面按鈕、API 呼叫
  • 鏈式：由另一個工作流觸發

你可以組合觸發方式（例如「每小時運行一次，或在上傳檔案時」）。

可以。Loop 提供Cron 風格排程：

  • 簡單排程：「每小時」、「每日上午 9 時」、「每週一」
  • 複雜排程：Cron 表達式（例如「0 9 * * 1-5」= 工作日上午 9 時）
  • 時區感知：以你的本地時區排程
  • 錯過運行：設定錯過運行的行為（跳過、運行一次、全部運行）

排程工作流適合定期任務（例如「每小時同步數據」、「每日產生報告」）。

可以。Loop 提供事件驅動觸發：

  • Webhook：透過 HTTP 請求觸發工作流
  • 檔案監視器：在檔案建立／修改時觸發工作流
  • ODW 代理事件：在代理完成任務時觸發工作流（例如「會議結束後」）
  • 數據庫事件：在數據庫記錄變更時觸發工作流

事件驅動工作流適合反應式自動化（例如「當客戶提交表單時處理數據」）。

可以。Loop 提供手動觸發：

  • 介面按鈕：在 Loop 主控台點擊按鈕運行工作流
  • API 呼叫：透過 REST API 觸發工作流
  • CLI：透過命令列介面觸發工作流

手動觸發適合按需任務（例如「立即產生報告」、「按需同步數據」）。

Loop 提供錯誤處理：

  • 重試邏輯：自動重試失敗的節點，可設定退避
  • 錯誤處理器：為每個節點定義自訂錯誤處理器
  • 備用工作流：如主工作流失敗，運行備用工作流
  • 警報：收到失敗通知（webhook、電郵、Slack）
  • 執行歷史：檢視失敗運行情況以診斷問題

目標：工作流失敗的平均復原時間 ≤2 分鐘。

可以。Loop 提供工作流控制：

  • 暫停：暫停正在運行的工作流（可恢復）
  • 停止：停止正在運行的工作流（不可恢復）
  • 取消：取消排程中的工作流
  • 停用：停用工作流（防止未來運行）

這些控制在排查問題、維護或停止失控工作流時非常有用。

可以。Loop 支援並行執行：

  • 並行分支：同時運行工作流的多個分支
  • 並行工作流：同時運行多個工作流
  • 資源限制：設定並發限制以防止資源耗盡

並行執行適合提升效能（例如「並行處理 100 份文件，而非逐份處理」）。

可以。Loop 提供等待節點：

  • 基於時間的等待：等待特定時長（例如「等待 5 分鐘」）
  • 基於事件的等待：等待外部事件（例如「等待 webhook 回調」）

05 · 部署

安全與私隱

8 個問題

安全。Loop 遵循安全最佳實踐建構：

  • 自託管：所有工作流數據留在你的基建上
  • 加密：靜止時 AES-256，傳輸時 TLS 1.3
  • 存取控制：RBAC、SSO（透過 Shield 整合）
  • 審計日誌：所有工作流執行的防篡改日誌（透過 Shield）
  • 開源：代碼可審核——你可以驗證沒有後門或數據外洩

Loop 為受監管行業而設。如果你能使用 Loop，就能通過合規審計。

不能。Loop 在你的基建上自託管。ODW.ai 完全無法存取你的工作流數據。

不同於雲端工作流工具（Zapier、n8n 雲端），Loop 不會將工作流數據傳送給第三方。一切留在你的伺服器上。

這對受監管行業至關重要。如果你的工作流工具正將數據傳送至雲端供應商，你就無法聲稱擁有數據主權。

可以。Loop 支援氣隙部署（無互聯網存取）：
